Builds directly on the v0.2 assessment: the encoder output `skill-pack:rust-engineering-general-v0.2` is a content-addressed signature inventory with 718 name-match validators; the compile path and provenance backbone are proven, capability uplift is not. ## Where v0.2 landed (the starting point)

`theseus.skill_encoder.v1` lowered 930 first-party `callable_signature_view` atoms into a buildable, content-addressed crate (`skill_encoder_validators`: a 1489-line `lib.rs` registry plus a `ValidatorReceipt` type plus 718 generated validator files). Every validator checks one thing, `candidate.name == "X"`, so it proves identity and provenance, not behavior; `validates_cosine_distance` checks the member is named `cosine_distance`, not that it computes one. The corpus was first-party only (the 21 external repos in `references/corpus.md` were never ingested), and the prose `SKILL.md` carried over from the seed is still doing all the teaching. The pack honestly marks itself `scanned (compiled, not yet held-out validated)` and every validator says `advisory; promotion requires the benchmark gate`.

Keep the wins: the compile path, content-addressing, the parent-hash provenance chains, the `scanned`/advisory trust ladder, and the `ValidatorReceipt` harness-runnable shape. v0.3 changes what gets lowered and emitted, not the backbone.

## Part 1: make the existing run real

### Lift real test bodies (identity to behavioral)

The decisive change. The lowering dialect today is `callable_signature_view`, so a validator can only compare signatures. Add a lowering that captures the body of code members that are tests (`#[test]` fns), and for those atoms emit the test body (or a call into it) as the validator instead of a name check. The descriptive-named validators (`validates_redcore_strict_acid_requires_aof_always`, `validates_hybrid_search_blends_vector_and_graph`, `validates_apply_skill_pack_writes_use_receipt`) are named after real tests in the THG suite, and their bodies already encode the behavioral contract. A lifted validator runs that contract, so a candidate passes only if the behavior holds, which is exactly what the held-out gate needs to measure. Confirm during implementation which source atoms are `#[test]` fns versus ordinary callables; the test atoms are the ones that carry a liftable behavioral assertion.

### Cut the signature inventory (curation)

930 lowered callables produced 718 validators including `default`, `new`, `fmt`, `clone`, `drop`, and a dozen duplicates of the same name across modules. That is an inventory of the codebase's surface, not a capability set, and it cuts against the corpus policy of representative slices over cloning everything. Keep validators where there is a behavior or contract to assert: tests, trait contracts, protocol and schema shapes, persistence edges, the things the domain map already names. Drop or down-rank bare constructors, `Default`/`Clone`/`Debug` impls, and trivial accessors. A curated set of a few dozen behavioral validators is worth more than 718 name checks and makes the pack legible.

### Split the corpus (general versus first-party)

`provenance.source_ref` is `rust-engineering-first-party-v1` and every symbol is a Theorem, RustyRed, or THG name; the external corpus was never ingested. So the current pack is a Theorem-substrate skill wearing a general-Rust name. Run the encoder twice: one pass over the external corpus (tokio, axum, tonic, serde, syn, candle, servo, and the rest of `corpus.md`) for a genuinely general `rust-engineering` pack, and one pass over the first-party code for a distinct `theorem-substrate` pack. Name them for what they are. The general pass is also what lets the pack help on code that is not yours.

### Then gate (scanned to canonical)

Only after the three changes above, run the held-out benchmark gate (baseline versus treatment over a Rust task set), attach real `SkillPackUseReceipt`s, and flip confidence from `scanned` to `canonical` only when treatment beats baseline. Until then the pack stays advisory, exactly as the encoder already marks it.

## Part 2: the better unit, rule/property packs

The v0.2 failure and the Part 1 fix both point at one thing: the unit matters. v0.2's unit was the code member, which can only yield identity. A test body is a better unit, but it is still tied to one specific behavior in one corpus. The most valuable and most reusable unit is a rule: a named, checkable property that should hold across code, with a validator that runs against any candidate, not just the symbol it came from.

This is the sharper articulation of "general code correctness" and "repository structure": they are two ruleset packs, and a ruleset pack is a learned linter the agent carries.

### Code-correctness ruleset

Named invariants over code, each with a runnable checker. For example: no `unwrap`/`expect` in non-test library code; public functions over enums are total (every variant handled); errors are returned, not swallowed; operations documented as idempotent stay idempotent under a property test; serialization round-trips (for all `x`, `decode(encode(x)) == x`); inputs are bounded (no unbounded allocation driven by an untrusted size). Static rules are AST checkers (`syn` or `ast-grep` style, which the corpus already contains). Behavioral rules are property tests (`proptest` style). The validator is the checker, and it passes or fails on real candidate code.

### Repository-structure ruleset

Named invariants over a repo's shape, each with a checker. For example: the crate dependency graph is acyclic; a manifest edge exists for every cross-crate import (the exact anti-pattern the `SKILL.md` already calls out); tests are co-located with the crate they cover; generated code is segregated from hand-written code; layering is respected (a `core` crate has no dependency on a `runtime` crate); workspace members are declared. These are filesystem plus manifest plus import-graph checks, cheap to run and high signal.

### Why this is the right target

A ruleset pack fixes everything v0.2 got wrong at once. It generalizes, because a rule is not tied to a corpus symbol, so it works on any repo and the first-party-only problem disappears. Its validators are real, because each rule is a checker that runs and can fail, so the held-out gate has something to measure and `canonical` means something. It is curated and small, a few dozen rules rather than hundreds of stubs. And it is actionable at apply time, because "check your code against these correctness and structure rules" is capability, where "a function named `cosine_distance` exists" is not.

### Two ways to source the rules

Mine them or author them, and you will want both. Mining: the clustering step that was supposed to find patterns can find invariants instead of enumerating members. Cluster the corpus's tests and members, and for each cluster the rule is the shared property behind the instances (every `GraphStore` impl has a durable-reopen test becomes the rule "storage types must prove durable reopen"). Authoring: many correctness and structure rules are known best practice, so author the rule plus its checker and use the corpus to confirm the rule holds or to surface violations, which also seeds the rule's initial fitness. Either way each rule is a content-addressed artifact with a checker validator, and `UseReceipt`s record which rules catch real issues, so the fitness loop promotes the rules that matter and retires the ones that do not, the same epistemic pattern as the anti-conspiracy cluster-fitness work.

## Tooling that makes the encoding more powerful

The v0.2 lowering is shallow (`callable_signature_view` is signatures only), which is why a validator could only match names. Three tooling layers raise the ceiling, and they are the build options for Parts 1 and 2. None of them is MLIR: MLIR is a compiler framework for transforming programs toward execution, and this encoder analyzes code into knowledge, so its multi-level-IR machinery is the wrong fit here.

Semantic extraction raises what a rule can express. Move from a signature view to a semantic view. Two mature paths: rust-analyzer's semantic layer, which already does name resolution, type inference, and MIR lowering behind an object-oriented HIR API and is incremental by default; or rustc's MIR through Charon, where Rust's harder semantics (moves, copies, drops, control flow, pattern lowering) become explicit. Either one takes an atom from "a function named X" to "what it takes, returns, and calls, and whether it can panic," which is what correctness rules need in order to check anything real.

A checker engine turns a rule into a runnable validator. ast-grep handles AST-structural rules by structure rather than text and is already in the corpus, so it is the fit for syntactic rules like "no unwrap in library code" or "manifest edge matches import." dylint runs custom Clippy-style lints from a dynamic library with full rustc access, for rules that need types or MIR rather than syntax. proptest covers the behavioral rules, round-trips and idempotency. A rule's validator is one of these checkers, not a name comparison.

Code-as-facts, rules-as-queries is the structural home for the ruleset packs. This is the CodeQL and Glean model: represent the corpus as a fact database and let a rule be a query over it. The RustyRed/THG graph is already most of a code-fact database, so the natural shape is to lower code into facts (symbols, types, calls, imports, test-covers edges) and let a rule be a graph query plus a checker, with `UseReceipt`s driving the fitness loop. This is where the Part 2 ruleset packs should live.
