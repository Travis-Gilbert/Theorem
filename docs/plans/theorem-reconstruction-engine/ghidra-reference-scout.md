# Ghidra Reference Scout

## Read

This pass intentionally looked past the obvious loader/disassembler/decompiler shape. The interesting thing in Ghidra is an analysis ecology around a durable program object:

- declarative language packs and compiler specs
- a Program database split into memory, symbols, references, functions, code, data types, relocations, and source maps
- event-driven analyzer scheduling over byte, instruction, function, signature, and data changes
- p-code as a finite semantic vocabulary for many processors
- function-start byte-pattern mining
- FunctionID library recognition
- BSim decompiler-derived similarity vectors
- VersionTracking match, confidence, markup, and precondition receipts
- data-type dependency ordering for structs, unions, arrays, typedefs, pointers, and function definitions
- fill-out-structure and pointer-relative recovery that follows p-code loads/stores/calls back to structure roots, offsets, and bitfields
- stack-frame recovery with local/parameter/save sections, growth direction, parameter offset, return-address offset, and stack-variable ordering
- stack-pointer flow and callsite extra-pop recovery that turns unknown callee stack effects into solver-backed ABI facts
- decompiler warnings, header comments, bookmarks, and export failures that turn analysis uncertainty into durable diagnostics
- high symbols, high variables, varnode instances, and ordered variable storage
- ParamID input/output parameter measures that rank candidate storage by data-flow evidence, model names, extra-pop, and prototype-only recovery
- p-code injection payloads, callfixups, callotherfixups, call mechanisms, executable p-code, userop mapping, and injection metadata that replace opaque calls with explicit semantics
- p-code emulation as a composable machine over value domains, callbacks, injected p-code, shared/thread-local state, trace writers, and taint/symbolic auxiliary domains
- flow overrides, primary override references, fallthrough overrides, localized prototype overrides, jump-table override namespaces, and effective-extra-pop stack delta overrides that record where analysis was intentionally forced
- cover/intersection-aware variable merging, heritage SSA passes, MULTIEQUAL/INDIRECT forced merges, speculative datatype merges, and COPY-trim split preservation
- decompiler action/rule pipelines, repeated action pools, opcode-indexed local rewrite rules, per-rule test/apply counters, restarts, and action group paths
- jump-table and switch-dispatch recovery with case labels, load tables, branch references, and default handling
- equates that bind scalar constants to names, enum identities, display formats, operand indexes, and dynamic-hash references
- external locations and function thunks that separate imported API boundaries from local implementation work
- debugger traces as time-indexed Program views
- auxiliary p-code domains such as taint

Theorem should not treat Ghidra only as a decompiler to imitate. It should treat Ghidra as a reference architecture for an always-on analysis substrate, then move the durable state into GraphStore and the semantic authority into learned/default scorers.

## The Ambitious Shape

The goal should not be "build a Rust decompiler." The more valuable goal is:

```text
Binary/source/runtime/docs
  -> normalized ProgramGraph facts
  -> scheduled analysis passes
  -> learned and symbolic recovery
  -> match/drift/version receipts
  -> engineering obligations and validators
  -> agent patch/review/test loop
```

This makes Ghidra a reference/oracle, not the center of the product. Theorem becomes the always-on analysis memory and engineering compiler.

## High-Leverage Ports

1. ProgramGraph ontology
   - Add graph node and edge contracts matching Ghidra's Program managers: MemoryBlock, CodeUnit, Instruction, Function, Symbol, Namespace, External, Reference, DataType, Relocation, SourceFile, Bookmark, Context, Property.
   - Preserve authority layers: observed, derived, hypothesis, instruction, validated.
   - Initial native port landed in `rustyred-thg-binformat` as `ProgramGraph`/`ProgramManager`: stable program root IDs, Ghidra ProgramDB manager order for memory/code/symbol/namespace/function/external/reference/data/equate/bookmark/context/property/tree/relocation/source-file managers, Theorem string/entrypoint manager extensions, GraphStore manager nodes, and manager-to-fact edges.

2. Analysis pass scheduler
   - Generalize current program-analysis run contracts into a scheduler for changed graph regions.
   - Pass metadata should include type, priority, applicability, default enablement, inputs, outputs, options, receipts, and re-run triggers.
   - Initial native port landed in `rustyred-thg-code` as `AnalysisPassSpec`, `AnalysisChangeSet`, and `AnalysisWorkItem`: Ghidra-style analyzer types for byte/instruction/function/function-modifier/function-signature/data passes, enabled/applicability filtering, deterministic changed-range coalescing, priority ordering, input hashes, scheduler work IDs, and GraphStore `AnalysisPassSpec`/`AnalysisWorkItem` nodes connected to `ProgramAnalysisRun`.

3. Language pack ingestion
   - Keep iced-x86 as a practical fast lane, but start a SLEIGH/p-code compatibility lane.
   - First cut can import Ghidra-generated p-code fixtures rather than compiling SLEIGH in Rust on day one.
   - Initial native port landed in `rustyred-thg-binformat` as Ghidra-style `ProgramLanguageSpec`, `ProgramProcessorSpec`, and `ProgramCompilerSpec` facts: x86-64 language id `x86:LE:64:default`, SLEIGH/pspec/cspec file links, compiler-spec aliases, external tool names for GNU/IDA/DWARF/Golang/QEMU, processor context defaults, register-group summaries, GCC data organization, ABI prototype names, and ProgramGraph language/compiler/processor edges.
   - Ghidra XML pack ingestion landed in `rustyred-thg-binformat` as `ghidra_language_specs_from_pack` and `ghidra_language_specs_from_directory`: Theorem now parses `.ldefs`, `.pspec`, and `.cspec` sources directly into language, processor, and compiler facts, including external tool aliases, processor properties, context/tracked values, register groups, stack pointer, default ABI prototype, data organization, prototype names, and callfixup p-code bodies plus body hashes. This moves the lane from a hand-coded x86 snapshot toward source-grounded Ghidra processor-pack ingestion.
   - Ghidra callfixup and callotherfixup payload parsing now preserves injection-specific fields in `rustyred-thg-binformat`: `<pcode>` `paramshift`, `dynamic`, and `incidentalcopy` attributes plus `<input>`/`<output>` parameter names and byte sizes are captured on `ProgramCompilerCallFixup` and `ProgramCompilerCallOtherFixup`, matching Ghidra's `InjectPayload` model. `callotherfixup` payloads preserve Ghidra's `targetop` field, which names the user-defined CALLOTHER op that the payload replaces.

4. P-code/THIR parity
   - Align THIR with the PcodeOp vocabulary enough to compare generated facts against Ghidra output.
   - Build fixture tests for branches, calls, loads/stores, pointer arithmetic, SSA joins, and switch recovery.
   - Initial native port landed in `rustyred-thg-lift` as `PcodeOpcode`/`PcodeFact`: Ghidra opcode IDs, THIR-to-p-code projection for copy/call/callind/branch/cbranch/branchind/return/unimplemented, GraphStore `PcodeFact` nodes, `STATEMENT_HAS_PCODE` edges, and focused opcode-ID tests.
   - Ghidra `PcodeOp` semantic metadata landed in `rustyred-thg-lift`: `PcodeOpcode` now exposes source-grounded opcode-id lookup, Ghidra-exact mnemonics including `INT2FLOAT`/`FLOAT2FLOAT`/`TRUNC` and template aliases (`BUILD`, `DELAY_SLOT`, `LABEL`, `CROSSBUILD`), opcode categories, control-flow/memory/internal flags, and Ghidra's commutative-op list. `PcodeFact` GraphStore nodes now carry `ghidra_mnemonic`, `opcode_category`, `is_commutative`, `is_control_flow`, `is_memory_access`, and `is_internal` so downstream recovery can query semantic p-code classes without lexical guessing.
   - Compiler-spec callfixups and callotherfixups now bridge into reconstruction contracts: `rustyred-thg-reconstruct` exposes `pcode_inject_contracts_from_program_compiler_specs` and `pcode_inject_contracts_from_program_language_specs`, lowering parsed Ghidra callfixups into `PcodeInjectContract`s with target names, param shift, dynamic/incidental-copy flags, input/output parameter ordering, body hashes, fallthrough standing, and opcode summaries. It also lowers parsed Ghidra callotherfixups into `PcodeInjectKind::CallOtherFixup` contracts with userop names/indexes, input/output parameters, body hashes, fallthrough standing, and CALLOTHER opcode summaries. `bind_pcode_inject_contracts_to_program_targets` mirrors Ghidra's callfixup target-name binding by matching recovered functions and binary symbols with exact, single-underscore, double-underscore, and `libID_conflict_` variants, then stamps function/symbol evidence so injected semantics flow into reconstruction obligations without manual assignment. `bind_callother_pcode_inject_contracts_to_userop_sites` mirrors Ghidra's `InjectedUserOp`/`CALLOTHERFIXUP_TYPE` attachment path by expanding global callotherfixup payloads onto concrete lifted `CALLOTHER` userop p-code facts, preserving statement, instruction, p-code fact, and original compiler-spec evidence. `flow_override_contracts_from_non_fallthrough_pcode_injects` mirrors Ghidra's non-fallthrough repair path: when a bound callfixup payload ends in `BRANCH`, `BRANCHIND`, or `RETURN`, typed call references to that target derive `CALL_RETURN` flow overrides with callsite, target, fallthrough, and evidence edges.

5. Function boundary and library recognition

1. Run the checked-in headless Ghidra exporter over small fixture binaries and refresh JSON matching `GhidraOracleExport` for:
   - loader facts
   - functions and basic blocks

2. Feed that JSON into the landed address-level oracle fact models.
3. Add parity tests comparing Theorem native output against oracle JSON for representative compiler/architecture cases.
4. Feed successful parity receipts into the learned scorer training stream.

That cut keeps ambition high without betting the whole project on immediately porting Ghidra's full decompiler. Theorem starts learning from Ghidra now, while Rust-native analysis grows module by module.

## Product Implication

This scout strengthens the earlier design direction:

- Theorem should do most of the reconstruction/engineering work.
- Agents should receive perception: obligations, drift, confidence, unknowns, source evidence, validators, and similar prior repairs.
- Tools remain for explicit actions; the high-value analysis should be ambient at run start.
- Learned scorers should decide semantic/component authority, with deterministic analyzers kept as evidence providers, candidates, validators, or fallback infrastructure.
