## Review Notes

- The new Ghidra jump-table slice is intentionally typed in `rustyred-thg-code` first, then lowered in `rustyred-thg-reconstruct`. That keeps GraphStore as the durable source of oracle facts.
- The new Ghidra equate slice follows the same pattern: `Program.getEquateTable()` is exported as observed oracle facts, then lowered into reconstruction obligations through the existing `EquateContract` lane.
- The new Ghidra external-linkage slice follows the same pattern: `ExternalLocation` plus thunk-chain facts are exported as observed oracle facts, then lowered into reconstruction obligations through the existing `ExternalLinkageContract` lane.
- The new Ghidra structure-field-access slice follows the same pattern: decompiler `ClangFieldToken` and `ClangBitFieldToken` facts are exported as observed oracle facts, then lowered into reconstruction obligations through the existing `StructureFieldAccessContract` lane.
- The new Ghidra high-variable slice follows the same pattern: `HighSymbol`, `HighVariable`, `VariableStorage`, and varnode instance facts are exported as observed oracle facts, then lowered into reconstruction obligations through the existing `HighVariableContract` lane.
- The new Ghidra stack-frame slice follows the same pattern: `Function.getStackFrame()`, `StackFrame`, stack variables, and `VariableStorage` facts are exported as observed oracle facts, then lowered into reconstruction obligations through the existing `StackFrameContract` lane.
- The new Ghidra ParamID slice follows the same pattern: `HighParamID` and `ParamMeasure` facts are exported as observed oracle facts, then lowered into reconstruction obligations through the existing `ParameterMeasureContract` lane. Java-exposed facts preserve model/extrapop/storage/type/rank; C++-internal call/source provenance stays null or zero until a deeper oracle bridge exists.
- The new Ghidra call-stack-effect slice follows the same pattern: `CallDepthChangeInfo`, callee purge size, and prototype extrapop facts are exported as observed oracle facts, then lowered into reconstruction obligations through the existing `CallStackEffectContract` lane.
- The new Ghidra function-prototype slice follows the same pattern: `Function.getPrototypeString(false, true)`, return parameter/type, parameter storage, calling convention, varargs/no-return/thunk/custom-storage, stack purge, and signature source are exported as observed oracle facts, then lowered into reconstruction obligations through the existing `FunctionPrototypeContract` lane.
- The new Ghidra analyzer-scheduler slice follows the same pattern at the run-contract level: `Analyzer`, `AnalyzerType`, `AnalysisPriority`, `AnalysisScheduler`, and `ProgramChangeSet` concepts become `AnalysisSchedulerState` nodes linked to `AnalysisPassSpec` nodes, so queued added/removed ranges and enabled/default/scheduled state are durable facts rather than hidden runner state.
- The new Ghidra annotation slice follows the same pattern: `Listing.getCommentAddressIterator`, `Listing.getComment`, and `BookmarkManager.getBookmarksIterator` facts are exported as observed annotation facts, then lowered into reconstruction uncertainty through the existing `DecompilerDiagnosticContract` lane.
- The new Ghidra VersionTracking receipt slice extends the existing `FunctionSemanticMatch` lane instead of replacing it: match sets, markup item statuses, and aggregate review receipts now store the same applied/rejected/don't-care/don't-know/error shape Ghidra uses, so agent-facing reconstruction can distinguish pending work, applied evidence, uncertainty, and failed application attempts.
- The new Ghidra VersionTracking association-transition slice ports the database-manager behavior around accepted matches: source/destination competing matches become blocked, rejected matches keep precedence, clears recompute whether a rejected or blocked match should become available, and transition receipts make those state changes graph-queryable.
- The direct Ghidra exporter now emits `jump_tables` from `HighFunction.getJumpTables()`. Local `analyzeHeadless` was not available on PATH in this session, so the checked-in `hello_switch` fixture is a contract/parsing/write-through proof; the real headless parity refresh remains the next oracle step.
- Deterministic logic here is infrastructure: address parsing, canonicalization, GraphStore write-through, and tests. Semantic authority still belongs to learned/scored reconstruction lanes as they come online.
