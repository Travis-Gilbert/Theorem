# Harness Rust Port (Theseus Python -> Theorem Rust)

Date: 2026-06-01
Status: Phase 1 kernel and Phase 2 replay/fork slice implemented in
`theorem-harness-core` and green against the Python reference corpus. Grounded
against the live Index-API checkout
(`../Index-API/apps/orchestrate/runtime/`), not the V3 architecture doc's prose.

## Goal

Port the Theseus harness from Python (canonical: `Index-API/apps/orchestrate/runtime/`)
into Rust, in Theorem. Theorem is "the Rust-native substrate spine"; the harness
is the stateful run controller the substrate-native browser, RustyWeb, and the
Commonplace room all sit on. Today it is Python; the user's directive is to get
the state machine and the rest of the harness into Rust here.

This is a relocation of an existing, working machine, not a greenfield build.
The V3 architecture doc (`commonplace-harness-architecture.md`) describes what
the harness already does and what V3 *adds on top* (room UI, MapArtifact,
charter, budget governor, Pairformer). Most of that build order lives in other
repos/surfaces. THIS plan ports the kernel those layers assume. It does not
implement the V3 build order.

## Canonical source (re-verify against live main; iteration continues)

Harness home: `Index-API/apps/orchestrate/runtime/`. ~48 modules, ~900KB Python.
The `apps/notebook/harness/` files are thin shims that re-export from here.

## Port surface, classified by coupling (grep-verified 2026-06-01)

The harness splits hard into a pure-logic spine and a Django/Memgraph/IO body.
Only the pure-logic spine is a straight port. The coupled body is a *rewrite*
against the Rust substrate (or stays Python until the Rust substrate side
exists), NOT a line-for-line port.

| `contracts.py` | 62KB | (none) | ~30 `@dataclass(frozen=True, slots=True)`. Deliberately Django-free (says so in its docstring). The run data model. |

| `replay.py` | 5KB | contracts, state_machine | replay/fork/compare. |
| `toolgraph.py` | 11KB | contracts | permission-aware toolkit catalog, selection, blocked-tool reasons. |
| `memory_contracts.py` | 13KB | (none) | Memory record dataclasses. |

### Django / Memgraph / IO coupled (rewrite, not port; later)

`contracts`-adjacent but IO-bound or huge: `coordination.py` (148KB),
`cross_surface_memory.py` (102KB), `orchestrate.py` (74KB), `service.py` (55KB),
`contracts.py` is fine but `storage.py`, `memory_records.py`, `memory_thg.py`,
`oracle.py`, `handoff_compiler.py`, `cua_*`, `workstream.py`,
`federation_transmit/receive.py` reach Django ORM, Memgraph, Redis, or HTTP.
These do not port straight. They become Rust services over the RustyRed
substrate, sequenced after the kernel and after the relevant substrate surface
exists in Rust. Do not promise them in Phase 1.

## The parity gate (the discipline that makes a two-agent port safe)

Per Theorem's byte-parity rule: a native port is not done until its parity gate
is green over the reference set.

1. Generate a fixture corpus of transition sequences from the Python reference:
   for each legal chain (and a set of illegal ones that must raise specific guard
   codes), record the input events and the resulting `state_hash_after` and any
   `HarnessGuardError` code. The shared corpus lives in
   `docs/plans/harness-rust-port/parity/fixtures.json`, generated by
   `docs/plans/harness-rust-port/parity/generate_fixtures.py`.
2. The Rust port replays each fixture and asserts byte-identical state hashes and
   identical guard-violation codes.
3. `state_hash.rs` must emit canonical JSON identical to Python's
   `json.dumps(..., sort_keys=True, separators=(',', ':'))` before sha256, or
   every hash diverges. This is the single highest-risk parity surface; pin it
   first with its own focused fixture set (hash a known run dict in both, compare).

## Phase 1 lanes (the Claude / Codex seam)

The kernel splits at the data-model / control-logic boundary, the same shape as
the SceneOS port. Both lanes meet at the `contracts` types and the
`hash_run_state` signature.

- **Lane A (types + tables + parity anchor):** `contracts.rs` (the ~30
  dataclasses -> serde structs with `to_dict`-equivalent canonical
  serialization), the static tables (`TRANSITION_REQUIREMENTS`,
  `ALLOWED_PREVIOUS_STATUSES`, the CACHE/ORACLE/CUA/CMH event-type sets,
  `TERMINAL_STATUSES`), and `state_hash.rs` (canonical JSON + sha256). Owns the
  parity anchor.
- **Lane B (executor + guards):** `apply_transition`, `_advance_run`,
  `_created_run`, `_reject_terminal_run`, and every `_validate_*` guard
  (`_validate_toolkit_payload`, `_validate_outcome_payload`,
  `_validate_learning_payload`, `_validate_federation_payload`,
  `_validate_cache_transition`, etc.), `HarnessTransitionError` /
  `HarnessGuardError`. Consumes Lane A's types.

Seam contract: Lane A publishes the `HarnessRunState` / `HarnessEventState` /
`HarnessTransitionInput` structs and `hash_run_state(&run) -> String`; Lane B
imports them and never reaches around them. The fixture corpus (built jointly,
from the Python reference) is the shared acceptance artifact.

## Phase 2.5

Done from the Rust build spec: `theorem-harness-runtime` starts the native
event-log layer. It stores each successful `theorem-harness-core` transition as:

- a `HarnessRun` node keyed by `harness:run:{run_id}` with the latest run state
  and state hash
- a `HarnessEvent` node keyed by `harness:event:{run_id}:{seq}` with the event
  receipt, before/after hashes, payload, and sequence
- `HARNESS_EVENT_OF` edges from events to their run
- `HARNESS_EVENT_NEXT` edges between contiguous event receipts

The runtime rejects event gaps and conflicting writes at the same `(run_id,
seq)`. This keeps the kernel pure and makes persistence a GraphStore-backed
runtime concern, matching the build spec without sacrificing the Python parity
gate. Tests cover both `InMemoryGraphStore` and reopening a `RedCoreGraphStore`
after durable writes.

## Phase 3.1

Started, then exposed, the direct coordination channel in
`theorem_harness_runtime::coordination`. This first substrate slice persists
coordination-room membership, one live intent per `(room, actor)`, and durable
presence records directly into `GraphStore`, with `RedCoreGraphStore`
reopen-tests. The second slice adds append-only direct messages, `@actor`
mention parsing, `COORDINATION_MENTIONS` graph edges, room message reads, and
consume-on-read inbox semantics. It mirrors the currently useful
room/intent/presence/messages shape while removing Django/cache as the only
durable path.

The first exposure step is implemented in `rustyred-thg-mcp`: native
`coordination_room`, `presence`, `coordination_intent`, `coordinate`,
`mentions`, `read_intents_for_room`, and `read_messages_for_room` tools now use
the Rust runtime coordination data model and persist through the MCP backend
GraphStore. Read tools are listed in read-only mode; write tools appear only
when MCP read/write mode is enabled. The MCP test suite covers a full
room/presence/intent/message/mention-consume/readback round trip against a
shared fixture store.

Not covered yet: event/decision/tension/reflection records, context injection
into the channel, contribution capture, and permission/cost hooks. Those remain
the rest of the direct-coordination module.

The HTTP transport exposure now lives in `apps/theorem-harness-server` over
`theorem-harness-runtime`. It serves run list/detail plus the native
coordination read contracts:

- `GET /harness/rooms/{room_id}` for room membership/task state.
- `GET /harness/rooms/{room_id}/presence` for participant freshness.
- `GET /harness/rooms/{room_id}/intents` for live actor claims.
- `GET /harness/actors/{actor}/mentions` for the actor inbox, with optional
  `consume=true`.

Each endpoint accepts `tenant` or `tenant_slug` as a query parameter and reads
from the same `RedCoreGraphStore` as the run transport.

Remaining runtime/substrate work beyond that: context IO retrieval, substrate
adapters, map repository writes, affordance execution wrappers, charter
compiler, budget governor, compaction, trace exports, adapter bridges, and
Pairformer control. Each gets its own plan slice when reached.

## Honest scope note

The full harness is ~900KB / ~48 modules. This slice ports the pure-logic spine
(Units 1-2 plus the small pure helpers, ~123KB of the 900KB) with parity gates,
and explicitly defers the Django/Memgraph-coupled body (coordination,
cross_surface_memory, orchestrate, service, the memory/oracle/handoff/toolgraph/
cua IO modules) as rewrites against the Rust substrate, not straight ports.
"Port the harness" is a multi-phase effort; the completed kernel proves the
approach end-to-end with a green parity gate.
