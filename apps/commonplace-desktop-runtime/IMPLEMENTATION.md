# commonplace-desktop-runtime — implementation plan

Builds the two handoffs as one system, in this order: (A) notify-backed ambient layer, (B) phone-controlled local instance, then the agent co-presence layer. Worktree `ambient-coedit` (branch same), isolated from Codex's reconstruction PR on `main`.

## Evidence map (reverse-engineered, pinned)

notify-rs (resolved by cargo): `notify` 8.2.0, `notify-debouncer-full` 0.5.0, `ignore` 0.4.26. Core API used:
- `new_debouncer(timeout, None, tx)` where `tx: std::sync::mpsc::Sender<DebounceEventResult>` (a `Sender` impls `DebounceEventHandler`). `DebounceEventResult = Result<Vec<DebouncedEvent>, Vec<Error>>`; `DebouncedEvent` derefs to `notify::Event`.
- `Debouncer::watch(&mut self, path, RecursiveMode::Recursive)`.
- `Event { kind: EventKind, paths: Vec<PathBuf> }`; `EventKind::{Create,Modify,Remove,Access,Any,Other}`.
- `ignore::gitignore::GitignoreBuilder::{new, add(&path)->Option<Error>, add_line(None, line), build}`; `Gitignore::matched(path, is_dir).is_ignore()`.
- The debouncer already coalesces rename/create/modify/remove + dedups — so this crate only maps a *settled* batch.

Landing-spot reality vs handoff:
- `crates/commonplace-desktop-runtime` did NOT exist. Built here as a **standalone crate under `apps/`** (own Cargo root, bare `[workspace]`), like `apps/desktop` / `apps/rustyred-embedded`, to avoid editing `rustyredcore_THG/Cargo.toml` `[workspace]` (which Codex has uncommitted + is PR-ing). Decision: where-code-lives is the builder's call.
- `crates/commonplace` exists: `IngestPipeline::{ingest, ingest_batch, ingest_routed}<S,B>`, `IngestInput` (+builder), `Embedder`/`DeterministicEmbedder`, `BlobStore`/`InMemoryBlobStore`, `content_hash`. This is the slice-2 ingest path.
- `theorem-harness-core/src/agent_binding.rs` exists — the action tiers for Part B authorization.
- Durable store: `rustyred-thg-core` `RedCoreGraphStore` + `DiskObjectStore` (the sidecar's graph + blobs).

## Canonical-git boundary (invariant for the whole of Part A)
The watcher is READ-ONLY on the tree. It never writes tracked files. RustyRed state lives in the sidecar (`<root>/.rustyred`, always ignored). Ingest + ambient passes write provenance into the sidecar graph, never back into the tree.

- [x] **B1 — authenticated control endpoint + device pairing (revocable, scoped per device)**. DONE.
  - **`src/pairing.rs` — `DevicePairing` registry.** Device-scoped, revocable, durable credential store. API:
    - `DevicePairing::open(sidecar_dir: &Path) -> Result<DevicePairing>` (cloneable `Arc`-backed handle).
    - `pair_device(label) -> Result<PairingResult>` issues a per-device id + bearer token; `PairingResult { device: DeviceSummary, token: String }` returns the raw token ONCE.
    - `verify(presented: &str) -> VerifyOutcome` (constant-time; `Authorized`/`Rejected`).
    - `list_devices() -> Vec<DeviceSummary>` (secret-free, includes revoked for audit).
    - `revoke_device(device_id) -> Result<bool>` (idempotent; `false` only if the id is unknown).
    - `device_count() -> usize`.
    - `PairedDevice { device_id, label, secret_hash, paired_at_ms, revoked, revoked_at_ms }`; `DeviceSummary` is the secret-free projection (drops `secret_hash` entirely).
  - **Security properties (the trust boundary):**
    - **Hashed at rest:** only the hex SHA-256 of the token is persisted (`secret_hash`); the raw secret is returned once and never stored (test `raw_secret_is_never_persisted` greps the file).
    - **Constant-time compare:** token-hash and pairing-code comparison both go through `constant_time_str_eq` (`subtle::ConstantTimeEq`). Unknown/revoked devices are compared against a fixed-length placeholder hash so the "device exists" branch is not a timing oracle.
    - **Token format:** `<device_id>.<hex-secret>`; the public device-id prefix lets `verify` look up exactly one candidate (then still compare the secret in constant time) rather than scanning.
    - **CSPRNG entropy:** `getrandom::fill` (OS CSPRNG); device secret = 32 bytes, pairing code = 8 bytes, both hex. A CSPRNG failure is an error, never a weak token.
  - **Durability choice (noted):** a serde-json file at **`<sidecar>/control/devices.json`**, NOT a second `RedCoreGraphStore`. Reason: the ingest sink already holds a process-level directory lock on the sidecar `graph` dir (see the spawn_watcher test comment); a second graph handle would contend for it, and the registry is a tiny list. Writes are atomic (temp file + rename) so a crash mid-write keeps the prior good file. A corrupt/missing-on-open file is a hard error (not a silent reset) so devices are never silently dropped. Survives reopen: test `registry_survives_reopen`.
  - **`src/control.rs` — loopback control endpoint (axum).** `ControlState::new(DevicePairing, pairing_code, RunRegistry)` (the `RunRegistry` arg landed with B3); `build_control_router(ControlState) -> axum::Router` (testable core); `serve_control(ControlState, port) -> Result<ControlServer>` binds **`127.0.0.1` only** (`LOOPBACK` const; ephemeral port via `0`), serves on a tokio task, `ControlServer::{local_addr, shutdown}`. `generate_pairing_code() -> Result<String>` makes the one-time code.
    - **Routes:** `GET /healthz` (unauthenticated liveness); `POST /pair` (gated by the one-time pairing code in the `x-pairing-code` header -> `403` without it; issues a token); `GET /v1/status`, `GET /v1/devices`, `POST /v1/devices/revoke` (all authenticated).
    - **Auth enforcement:** the `DeviceAuth` axum extractor (`FromRequestParts`) reads `authorization: Bearer <token>`, calls `DevicePairing::verify`, and returns `401` on missing/garbage/wrong-scheme/revoked before the handler runs. Putting `DeviceAuth` in a handler signature is what gates `/v1/*`.
    - **Loopback enforced:** bind address is always `LOOPBACK`, never `0.0.0.0`; cross-device reach is the relay's job (LATER). No secrets are logged (pair/revoke errors log a generic message; internal errors return a generic body).
  - **lib.rs re-exports:** `DevicePairing, DeviceSummary, PairedDevice, PairingResult, VerifyOutcome` and `build_control_router, serve_control, generate_pairing_code, ControlServer, ControlState, DeviceAuth, LOOPBACK, PAIRING_CODE_HEADER`.

- [x] **Real `RunExecutor` (process-spawning executor)**. DONE. `src/process_executor.rs` -- `ProcessRunExecutor`, the production counterpart to `MockExecutor`: it executes a submitted run LOCALLY AGAINST THE WORKING TREE by spawning a REAL child process and streaming its output as run events.
  - **API:** `ProcessRunExecutor::shell(root)` (the default: runs each run's `intent` as `sh -c <intent>` in `cwd = root`) and `ProcessRunExecutor::with_command_factory(root, factory)` (a `CommandFactory` seam to plug in a claude/codex command builder). `root()` accessor. Re-exported from `lib.rs`: `ProcessRunExecutor`, `CommandFactory`, `ResolvedCommand`, `ShellCommandFactory`. Production wires the run channel with `RunRegistry::new(Arc::new(ProcessRunExecutor::shell(root)))`; tests keep `MockExecutor`.
  - **Stream-event mapping:** child stdout is read line-by-line -> each line is a `RunEvent::Trace`; child stderr is drained on its own thread (so a full stderr pipe can never deadlock the child) and surfaced as trailing `stderr: <line>` Traces; on child exit a `git diff --stat` over `root` is emitted as `RunEvent::Diff` (always emitted -- "no changes" / "diff unavailable (not a git working tree)" / "diff unavailable (git not found)" when there is nothing or no repo, so a run ALWAYS surfaces a diff summary); the terminal `RunEvent::Status` is `Done` (exit 0), `Failed` (nonzero exit / spawn error / empty command), or `Stopped` (cooperative cancel). The registry's `dispatch` already runs the executor on `spawn_blocking`, so the blocking process I/O never stalls the SSE reactor.
  - **How stop kills the process:** the run channel's `stop` sets the cooperative cancel flag the executor reads via `RunEventSink::is_cancelled`. Because a blocking `read_line` could not observe that flag while a SILENT child (e.g. `sleep 120`) produces no output, stdout is read on a dedicated reader thread that pushes lines down an mpsc channel; the executor's main loop polls that channel on a 50ms timeout and re-checks the cancel flag each tick. On a stop it calls `Child::kill()` (a real SIGKILL), `wait()`s to reap the child, and returns `RunState::Stopped` (no Diff -- the work was interrupted). So a stop both ends the run AND terminates the spawned process. An already-cancelled run never spawns.

Overlap is line/col range intersection on the flat `(line, col)` ordering via `ranges_overlap` (closed spans: two ranges overlap unless one ends strictly before the other starts; a zero-width insertion point inside a span overlaps -- the conservative "warn" choice). `ActorId` is derived from a stable agent label via `ActorId::from_label`, so two processes that pass the same label (`"claude-code"`, `"codex"`) share one presence identity.

### Plugin-hook integration contract (deliverable 3, DEFERRED to the plugin/CLI repo)
The edit hooks live in the Claude Code / Codex plugin layer (separate repo); there is no oracle for them in this worktree. The substrate they call is COMPLETE and verified here. The exact contract a hook implements (so the plugin work is a clean pickup):

**Identity.** Each agent process picks ONE stable `actor` label for its whole lifetime: `"claude-code"` for Claude Code, `"codex"` for Codex (a human surface would use a person label with `kind:"human"`). The same label across calls = the same presence identity (the server hashes it to an `ActorId`). `path` is the workspace-relative file path the agent uses; the registry treats it as an opaque key, so the hook must be consistent (pick repo-relative and stick to it).

**Auth + endpoint.** The hook holds the device bearer token (paired once via `POST /pair` with the one-time pairing code) and the local base URL (the loopback control endpoint, e.g. `http://127.0.0.1:<port>`). Every call carries `Authorization: Bearer <token>`.

**On `PreToolUse(Edit)`** (before any write), in this order:
1. `POST /v1/presence` with `{actor, path, line, col, label, kind:"agent"}` -- announce the cursor/focus (the line/col of the intended edit, or the file head if unknown).
2. `POST /v1/presence/footprint` with `{actor, path, range:{start_line,start_col,end_line,end_col}, summary}` -- set the INTENDED edit range (`summary` = a short description of the change).
3. `POST /v1/presence/would-overlap` with `{actor, path, intended:{...same range...}}` -- read peers' overlapping pending edits. A NON-empty `overlaps` array means a peer announced an intent to touch overlapping lines.
   - **Overlap policy (the hook's call; default warn-and-proceed):** bytes still merge via git + Claude Code's content-anchored `Edit` + the file freshness-guard, so the safe default is to WARN (surface the peer's `actor` + `summary` + `range`) and proceed. Serialize-on-overlap (wait/retry) is the stricter option. A tier-2/3 SEMANTIC-ownership conflict escalates to the phone authorization surface (`/v1/runs` approve tiers) -- out of scope for the awareness layer.

**On `PostToolUse(Edit)`** (after the write completes):
4. `DELETE /v1/presence/footprint` with `{actor, path}` -- clear the pending-edit footprint (idempotent; a clear with no prior set is harmless). The presence announce can persist (it is the cursor) or be cleared when the file closes via `clear_presence` (no dedicated route yet -- see deferred below).

**Ordering invariant:** announce -> set footprint -> would-overlap (pre); clear footprint (post). The would-overlap read MUST come after the caller's own set so re-running it is self-consistent (the caller's own footprint is excluded by `actor`, so seeing your own set does not self-flag).

## Local-first connection path (CommonPlace desktop webapp connects to THIS instance)
Goal: the CommonPlace desktop webapp connects to the local instance for commonplace data. Architecture decision (held, not deviated): the local instance is ONE process serving DATA + control + ambient over ONE in-process commonplace graph; a second process opening the same `RedCoreGraphStore` AOF would conflict. So the data is exposed over the EXISTING control endpoint, over the SAME graph the `CommonplaceIngestSink` maintains.

- [x] **One-graph shared handle** (`src/sink.rs`): `SharedSink` wraps `Arc<Mutex<CommonplaceIngestSink>>` (`SidecarCommonplaceStore = Commonplace<RedCoreGraphStore, DiskObjectStore>`). It IS a `ChangeSink` (the watcher writes through it) AND exposes read accessors (the control endpoint reads through a CLONE of the same handle), so a watcher ingest is immediately visible to a read with no second AOF handle. `AmbientRuntime` now holds the sink as a `SharedSink` (`AmbientRuntime::from_shared` / `shared_sink()`), and `run_cycle` locks it ONCE per cycle so an ingest + its passes commit atomically with respect to a concurrent read (a read never sees a half-ingested cycle). `lib.rs` gains `spawn_ambient_with_data(config) -> (WatchHandle, SharedSink)`: start the watcher and get the read handle onto the same graph in one call (`spawn_ambient` delegates to it). The watcher stays read-only on the tree; reads are read-only on the graph (this slice does NOT mutate the graph over HTTP).

- [x] **Data read routes** (`src/control.rs`, all behind the existing `DeviceAuth`, loopback already). Grounded against commonplace's PUBLIC query API (read `crates/commonplace/src/store.rs` + `ingest.rs`; mirror what `apps/commonplace-api` uses), reading the sink's durable `Commonplace`:
  - `GET /v1/items?limit=` -> `SharedSink::list_items` over `Commonplace::all_items` (sorted newest-first by `updated_at_ms`, clamped to `[1,1000]`, default 100). Body `{ items: [ItemJson] }`.
  - `GET /v1/items/{id}` -> `Commonplace::get_item`. Body `{ item: ItemJson }`; `404` if absent.
  - `GET /v1/collections` -> `GraphStore::query_nodes(label = Collection)` (UFCS past the RedCore inherent-vs-trait shadow) + `Commonplace::get_collection` per id (mirrors commonplace-api's `collections` resolver exactly). Body `{ collections: [CollectionJson] }`.
  - `GET /v1/search?q=&k=` -> the REAL commonplace search `IngestPipeline::search(&commonplace, q, k)` (the engine vector index the ambient ingest populates, via `RedCoreGraphStore: EmbeddingGraphStore`), then `get_item` per hit (clamped to `[1,200]`, default 20). NOT an invented text scan. Body `{ hits: [{ item: ItemJson, score }] }`; blank `q` -> `400`.
  - **JSON shape**: `ItemJson` is a deliberate projection of commonplace `Item` -- `{ id, kind, title, text?, blob?{content_hash,byte_len,mime?}, source?, residency, tags, collections, classification?, created_at_ms, updated_at_ms }`. It DROPS the heavy embedding vector and never inlines a `File`'s bytes (it reports the content-addressed `blob` ref instead). `CollectionJson` = `{ id, name, kind, created_at_ms }`. `ControlState::with_data(SharedSink)` wires the routes; without a sink they return `503` (the endpoint is up but the data backend is not attached). Re-exported from `lib.rs`: `ItemJson`, `BlobRefJson`, `CollectionJson`, `ItemListResponse`, `ItemDetailResponse`, `CollectionListResponse`, `SearchHitJson`, `SearchResponse`.

- [x] **Local-access token** (`src/pairing.rs`): a same-machine webapp on loopback should NOT do the QR/pairing-code dance, so the instance auto-provisions a `local_access` device. `DevicePairing::ensure_local_access() -> LocalAccess { device_id, token }` mints ONE device labelled `local_access` THROUGH THE SAME `pair_device` path (so it verifies via the same `DeviceAuth`/`verify` path -- one auth model) and persists its token at `<sidecar>/control/local_access.json` (atomic temp+rename, like the device registry). It is idempotent + self-healing: a persisted token that still verifies is reused; a missing file or a revoked/removed device (token no longer verifies) triggers a fresh mint. `DevicePairing::local_access_token() -> Option<String>` is what the Tauri shell reads to hand the webapp its `Authorization: Bearer` token. Revoke it like any device via `revoke_device(device_id)` -- the token stops verifying immediately. The token file holds the RAW token on purpose (it is the credential the shell hands the webapp); it lives in the same gitignored sidecar control dir as the registry (same trust domain) and only authorizes the loopback endpoint. Re-exported: `LocalAccess`, `LOCAL_ACCESS_LABEL`.

- **Tests (this slice; both validation gates green; ALL prior tests still pass).** `control.rs` unit (3, `tower::oneshot` over a tempdir-backed sink): list/get/search over seeded items (incl. unknown-id 404 + blank-`q` 400); every `/v1/*` data route 401 without a token; data routes 503 with no sink wired. `pairing.rs` unit (4): local access provisions+persists+authorizes + is listed/labelled; idempotent + survives reopen; revocable + self-heals after revoke. `tests/data_routes.rs` HTTP acceptance (2, real loopback bind + reqwest): **the headline -- start the watcher via `spawn_ambient_with_data`, write a real file into the watched tree, and `GET /v1/items` + `GET /v1/search` over HTTP surface it (proving the watcher's write is visible to the read because they share ONE graph), with `/v1/items` 401 without a token**; the local-access token authorizes the data routes over HTTP with no pairing dance and stops working after `revoke_device`. `cargo test` (84 passed / 1 ignored across all binaries) + `cargo clippy --all-targets --no-deps -- -D warnings` both green.

### The desktop webapp local-first connection contract (cross-repo follow-up, documented not built)
